## Recommended Reading Order

1. Start with the `high-level architecture` diagram to understand the platform shape.
2. Read the `call runtime sequence` diagram to understand the real-time backend flow.
3. Use `backend-runtime.md` for service ownership, data boundaries, and operational intent.

## Architecture Principles

These documents follow a few rules on purpose:

- the telephony layer is shown first because it is the entry point to the system
- the Go backend is shown as the real-time edge because it owns call lifecycle and media handling
- the Python backend is shown as the AI runtime because it owns STT, policy, LLM orchestration, and TTS
- persistent storage and post-call processing are separated from the low-latency path
- future scale components are shown, but visually isolated from the MVP-critical flow

## What These Documents Are For

They are meant to answer:

- how a call enters the platform
- how the backend processes audio in real time
- where policy and safety decisions happen
- where state is stored during and after the call
- how the user interacts with the system outside the call

They are not meant to replace:

- low-level API contracts
- database schema documents
- deployment manifests
- provider-specific implementation notes
